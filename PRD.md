# 더서브온 AI 자가진단 대시보드 — MVP PRD (Claude Code 실행용)

> 이 문서는 **컨설턴트 2번이 Claude Code로 개발**하기 위한 제품 요구사항 문서(PRD)입니다.
> 짝 문서인 `PRD_더서브온_자가진단대시보드_Codex.md`(컨설턴트 1번, Codex 담당)와 **기능 요구사항이 100% 동일**합니다. 두 프로토타입 중 하나를 선택할 예정이므로, 비교 기준이 "기능의 많고 적음"이 아니라 "실행 품질"이 되도록 아래 범위를 임의로 늘리거나 줄이지 마십시오.
>
> 이 앱은 앞서 만든 "더서브온 AI 콘텐츠 팩토리"(1소스 멀티유즈 생성 도구)와는 **별개의 자매 제품**입니다. 콘텐츠 팩토리가 "만드는 도구"라면, 이 앱은 "측정하고 진단하는 도구"입니다.

## 1. 프로젝트 배경과 목표

더서브온(서브온 주식회사) AI 활용 컨설팅팀은 7/6~7/7 이틀간 사람이 직접 5종의 진단(노출 테스트, 키워드·경쟁 분석, 콘텐츠 감사, 채널 정합성, 전환 동선 점검)을 수행해 사전진단 보고서를 작성했습니다. 이 과정은 매우 유용했지만, 컨설팅이 끝난 뒤에는 더서브온 담당자가 매번 컨설턴트를 부를 수 없습니다.

이 웹앱은 **그 5종 진단의 방법과 판정 로직을 자동화 도구로 코드화**하여, 더서브온 담당자가 (예: 매달 1회) 자사 블로그 RSS 주소를 넣고 몇 가지 체크리스트만 채우면, 컨설턴트가 만든 것과 같은 논리의 "종합 진단 리포트"를 스스로 뽑아볼 수 있게 하는 것이 목표입니다.

**목표 우선순위:**
1. 더서브온이 **직접, 반복적으로, 스스로** 운영할 수 있는 완성도 (최우선)
2. 컨설팅팀이 사람이 손으로 했던 판정 로직을 정확히 재현 (신뢰성)
3. 시간이 되면 7/8 미팅에서 짧게 미리보기 데모 가능

## 2. 제품 개요

- **제품명(가칭):** 더서브온 AI 자가진단 대시보드 (SERVEON Self-Diagnosis Dashboard)
- **한 줄 정의:** 블로그 RSS 주소와 몇 가지 체크리스트만 입력하면, 컨설턴트가 쓰던 5대 진단 로직 그대로 온라인 노출·도달·전환 상태를 자동으로 점검해 주는 셀프 진단 도구
- **핵심 가치:** 컨설팅이 끝난 뒤에도 "우리 콘텐츠가 지금 어떤 상태인지"를 외부 도움 없이 스스로, 반복적으로 확인할 수 있다

## 3. 사용자와 사용 시나리오

| 사용자 | 시점 | 시나리오 |
|---|---|---|
| 더서브온 콘텐츠 담당자 (주 사용자) | 컨설팅 종료 후 매달 | 블로그 RSS 주소 입력 → 자동 콘텐츠 감사 실행 → 나머지 체크리스트(노출/키워드/채널/전환) 채움 → "진단 리포트 생성" → 지난달과 비교하며 개선 방향 확인 |
| 컨설턴트 (시연자) | 7/8 미팅 (선택적) | "이런 도구를 만들어 드릴 수 있습니다"라며 샘플 데이터로 전체 흐름을 짧게 시연 |

**핵심 사용 흐름 (해피 패스):**
```
① [자동] 블로그 RSS 주소 입력 → 최근 게시물 자동 수집·분류 (콘텐츠 감사, D3)
→ ② [반자동] 노출 테스트: 시스템이 만들어준 검색 링크를 열어 확인 후 결과 입력 (D1)
→ ③ [반자동] 키워드 데이터 입력 (D2)
→ ④ [반자동] 채널 체크리스트 입력 (D4)
→ ⑤ [반자동] 전환 동선 체크리스트 입력 (D5)
→ ⑥ "진단 리포트 생성" 버튼 → AI가 종합 판정표 + 우선순위 개선과제 + 확인 질문 생성
→ ⑦ 리포트 저장(이력) → 다음 달 진단과 비교
```

**왜 "반자동"인가:** 네이버 검색 순위나 SNS 인사이트를 완전 자동으로 긁어오는 것은 약관·API 제약이 있습니다(11장 참조). 그래서 시스템은 "확인할 링크를 만들어주고 사람이 눈으로 보고 결과만 입력"하는 방식을 택합니다. 이는 컨설팅팀이 실제로 사람이 검색해서 확인했던 방식과 동일한 논리이며, MVP 범위에서 가장 현실적입니다.

## 4. 기능 요구사항

### 4.1 MVP 범위 (반드시 구현)

| # | 기능 | 자동화 수준 | 설명 |
|---|---|---|---|
| F1 | 블로그 RSS 자동 수집 | 완전 자동 | RSS 주소 입력 → 서버가 fetch·파싱 → 최근 게시물(제목/발행일/본문 요약) 목록화 |
| F2 | 콘텐츠 자동 분류 | 완전 자동(AI) | 게시물별 제목 유형(질문형/체크리스트형/사례형/지역형/비교형), 콘텐츠 목적(노출/공감/신뢰/전환), 8대 카테고리 자동 태깅 |
| F3 | CTA 자동 탐지 | 완전 자동 | 문의/전화/카카오/상담/신청 키워드 포함 여부를 게시물마다 자동 카운트 |
| F4 | 발행 콘텐츠 안전 스캔 | 완전 자동(AI) | 이미 발행된 게시물 본문에 금지 표현(8장 표) 포함 여부 자동 스캔·하이라이트 |
| F5 | 노출 테스트 반자동 입력 | 반자동 | 시스템이 "제목검색"·"핵심키워드검색"용 네이버 모바일 검색 URL을 자동 생성 → 담당자가 새 탭에서 확인 후 결과(노출/미노출)를 체크박스로 입력 |
| F6 | 키워드 데이터 입력 | 반자동 | 키워드 3층 전략표(사전 제공, 편집 가능) + 검색량/경쟁도 수동 입력 필드 |
| F7 | 채널 정합성 체크리스트 | 반자동 | 채널별(블로그/플레이스/카카오/인스타/페북/유튜브/당근/카페) 운영 여부·최근 활동일 입력 → 자동으로 사분면 시각화 |
| F8 | 전환 동선 체크리스트 | 반자동 | 글→CTA→신청→완료 단계별 통과/차단 여부 입력 → 자동으로 퍼널 시각화 |
| F9 | 종합 진단 리포트 생성 | 완전 자동(AI) | F1~F8 입력을 8장 마스터 진단 프롬프트로 전달 → 종합 판정표, 한 문장 진단, 우선순위 개선과제(최대 5개), 다음 확인 질문 자동 생성 |
| F10 | 진단 이력 저장 | 완전 자동 | 생성된 리포트를 localStorage에 스냅샷으로 저장, 리스트로 열람 |
| F11 | 리포트 인쇄/내보내기 | 자동 | 브라우저 인쇄(Ctrl+P)로 PDF 저장 가능하도록 인쇄용 레이아웃(CSS `@media print`) 제공 |
| F12 | 샘플 진단 불러오기 | 자동 | API 호출 없이 사전 입력된 실제 사전진단 데이터(demo-assets 참조)를 즉시 불러와 전체 흐름을 한 번에 체험 — 온보딩 겸 데모 안전망 |

## 6. 데이터 모델

```typescript
type PostPurposeTag = "노출" | "공감" | "신뢰" | "전환";
type TitleFormat = "질문형" | "체크리스트형" | "사례형" | "지역형" | "비교형" | "기타";
type Category =
  | "병원동행_사례" | "보호자_고민_해결" | "검사_진료별_안내"
  | "계절성_건강_콘텐츠" | "서비스_이용_안내" | "기업복지_콘텐츠"
  | "공공_사회적가치" | "매니저_신뢰_콘텐츠";

interface AuditedPost {
  title: string;
  pubDate: string;
  excerpt: string;
  titleFormat: TitleFormat;
  purposeTag: PostPurposeTag;
  category: Category;
  hasCTA: boolean;
  ctaKeywordsFound: string[];       // 예: ["상담"]
  safetyFlags: { phrase: string; suggestion: string }[];
}

interface D1ExposureResult {
  checkedTitle: string;
  titleSearchUrl: string;
  keywordSearchUrl: string;
  titleFoundInSearch: boolean;      // 담당자가 확인 후 입력
  keywordTopPageFound: boolean;     // 담당자가 확인 후 입력
}

interface D2KeywordEntry {
  keyword: string;
  tier: "1층_핵심전환" | "2층_상황검사" | "3층_지역B2B";
  monthlySearchVolume?: number;     // 선택 입력
  competitionLevel?: "낮음" | "보통" | "높음";
}

interface D4ChannelEntry {
  channel: string;                  // 예: "네이버 블로그"
  isOperating: boolean;
  lastActivityDaysAgo: number;
  targetStrength: number;           // 0~10, 고정값 (8장 참조), 수정 가능
}

interface D5FunnelStep {
  stepName: string;                 // 예: "블로그 글 CTA 클릭"
  passed: boolean;                  // true=통과, false=차단
  note?: string;
}

interface DiagnosisReport {
  overallVerdict: string;
  verdictTable: { item: string; result: string; status: "정상" | "취약" | "주의" }[];
  contentPurposeBalance: { 노출: number; 공감: number; 신뢰: number; 전환: number; unbalancedTags: string[] };
  priorityActions: string[];        // 최대 5개
  safetyFlagsFound: { postTitle: string; phrase: string; suggestion: string }[];
  channelQuadrant: { channel: string; targetStrength: number; operationLevel: "높음" | "보통" | "낮음"; priority: boolean }[];
  followUpQuestions: string[];
}

interface DiagnosisSnapshot {
  id: string;
  createdAt: string;
  d1: D1ExposureResult[];
  d2: D2KeywordEntry[];
  d3: AuditedPost[];
  d4: D4ChannelEntry[];
  d5: D5FunnelStep[];
  report: DiagnosisReport;
}
```

로컬 스토리지 키: `serveon_diagnosis_snapshots` (배열, 최신이 앞)

## 7. API 명세

### `POST /api/fetch-rss`
**Request:** `{ "rssUrl": "https://rss.blog.naver.com/아이디.xml" }`
**Response:** `{ "posts": [{ "title": "...", "pubDate": "...", "excerpt": "..." }] }` (AI 호출 없이 서버에서 XML 파싱만 수행 — `fast-xml-parser` 등 사용)

### `POST /api/audit-posts`
**Request:** `{ "posts": [{ "title": "...", "excerpt": "..." }] }`
**Response:** `{ "auditedPosts": AuditedPost[] }` — 8장 "콘텐츠 감사 서브 프롬프트" 사용, 게시물별 자동 분류·CTA탐지·안전스캔

### `POST /api/generate-report`
**Request:**
```json
{
  "d1": "D1ExposureResult[]",
  "d2": "D2KeywordEntry[]",
  "d3": "AuditedPost[]",
  "d4": "D4ChannelEntry[]",
  "d5": "D5FunnelStep[]"
}
```
**Response:** `DiagnosisReport` (6장 참조) — 8장 "종합 진단 마스터 프롬프트" 사용

**공통 에러 응답:** `{ "error": "사용자에게 보여줄 한글 에러 메시지" }`

**환경변수:**
```
ANTHROPIC_API_KEY=       # .env.local, 절대 커밋 금지
ANTHROPIC_MODEL=claude-sonnet-5
```

## 8. AI 프롬프트 사양

### 8.1 콘텐츠 감사 서브 프롬프트 (`/api/audit-posts`에서 사용)


### 8.2 종합 진단 마스터 프롬프트 (`/api/generate-report`에서 사용)

**중요:** 이 프롬프트는 컨설팅팀이 사람이 직접 수행한 사전진단과 동일한 판정 논리를 코드화한 것입니다. 판정 기준 수치를 임의로 바꾸지 마십시오.

```
당신은 더서브온(서브온 주식회사)의 온라인 마케팅 진단 전문가입니다.
아래 판정 기준을 그대로 적용해 객관적인 진단 리포트를 작성하십시오.
확인되지 않은 사실을 단정하지 말고, 데이터가 부족하면 "추가 확인 필요"라고 명시하십시오.

[회사 정보]
- 서비스: 병원동행(이동·접수·진료·검사·수납·약국·귀가 전 과정 동행), 재가복지센터(방문요양),
  기업복지 동행, 서울시 1인가구 동행서비스
- 상담 채널: 전화 1533-1683, 카카오 채널 "서브온_병원동행", 홈페이지 https://www.theserveon.com/

[D1 노출 테스트 판정 기준]
- 입력된 title 검색 결과 중 titleFoundInSearch=true 비율이 70% 이상 → "색인 정상"
- 70% 미만 → "색인 상태 확인 필요(저품질 의심)"
- keywordTopPageFound=true가 1건 이상 → "키워드 경쟁력 있음", 0건 → "키워드 경쟁 열세"

[D2 키워드 판정 기준]
- 입력된 키워드를 3층(1층_핵심전환/2층_상황검사/3층_지역B2B)으로 확인하고,
  2층 키워드 중 검색량이 있고 경쟁도가 "낮음"·"보통"인 키워드를 "단기 승부처"로 최소 1개 추천

[D3 콘텐츠 감사 판정 기준]
- 게시물 pubDate 기준 최근 1개월 발행 수 3건 이상 → "발행량은 문제 아님", 미만 → "발행량 부족"
- hasCTA=true 비율이 50% 미만 → "전환 유도 부족"
- purposeTag 분포를 계산해 권장 배분(노출40%/공감25%/신뢰25%/전환10%)과 비교, 10%p 이상 차이나는
  태그를 contentPurposeBalance.unbalancedTags에 기록
- safetyFlags가 하나라도 있으면 safetyFlagsFound에 postTitle과 함께 모두 기록

[D4 채널 정합성 판정 기준]
- lastActivityDaysAgo ≤ 30 → operationLevel="높음", ≤ 90 → "보통", 그 외 → "낮음"
- targetStrength ≥ 7 이고 operationLevel="낮음"인 채널을 priority=true로 표시 (최우선 보강 대상)

[D5 전환 동선 판정 기준]
- 입력된 단계 중 passed=false가 하나라도 있으면 해당 퍼널을 "병목"으로 판정하고
  verdictTable에 어느 단계인지 명시
- 모두 passed=true면 "양호"로 판정

[출력 형식 — JSON만 출력, 다른 텍스트 금지]
{
  "overallVerdict": "종합 진단을 한 문단으로, 단정 대신 '~로 보입니다' 어투 사용",
  "verdictTable": [
    {"item": "D1 노출", "result": "...", "status": "정상|취약|주의"},
    {"item": "D2 키워드", "result": "...", "status": "정상|취약|주의"},
    {"item": "D3 콘텐츠", "result": "...", "status": "정상|취약|주의"},
    {"item": "D4 채널", "result": "...", "status": "정상|취약|주의"},
    {"item": "D5 전환동선", "result": "...", "status": "정상|취약|주의"}
  ],
  "contentPurposeBalance": {"노출": 0.0, "공감": 0.0, "신뢰": 0.0, "전환": 0.0, "unbalancedTags": ["..."]},
  "priorityActions": ["...", "...", "...", "...", "..."],
  "safetyFlagsFound": [{"postTitle": "...", "phrase": "...", "suggestion": "..."}],
  "channelQuadrant": [{"channel": "...", "targetStrength": 0.0, "operationLevel": "높음|보통|낮음", "priority": true|false}],
  "followUpQuestions": ["...", "..."]
}
```

## 9. 안전장치 요구사항

- `safetyFlagsFound`는 **이미 발행되어 있는** 콘텐츠에서 발견된 위험 표현이므로, UI에서 "즉시 수정 권장" 딱지와 함께 눈에 띄게(붉은색 계열) 표시하십시오.
- 리포트 최상단에 "이 진단은 참고용이며, 중요한 의사결정 전에는 실제 데이터를 재확인하십시오"라는 고정 안내 문구를 표시하십시오.
- D1(노출 테스트)과 D5(전환 동선)는 사람이 직접 확인한 값을 신뢰하는 구조이므로, 담당자가 입력을 생략한 항목은 리포트에서 "미입력 — 진단 제외"로 명확히 구분해 표시하십시오(빈 값을 마치 실측한 것처럼 보여주면 안 됨).

## 11. 기술 스택 권장 (Claude Code 특화) 및 자동화 범위 제약 고지

Codex 버전과 **동일한 스택**을 사용해야 공정한 비교가 됩니다.

- **프레임워크:** Next.js 14+ (App Router) + TypeScript
- **스타일:** Tailwind CSS
- **AI 연동:** `@anthropic-ai/sdk`, 서버 사이드 Route Handler에서만 호출
- **RSS 파싱:** `fast-xml-parser` (또는 동등한 경량 XML 파서)
- **차트:** `recharts` (파이차트, 막대차트, 산점도 모두 지원)
- **저장소:** 브라우저 `localStorage`
- **배포:** Vercel

**중요 — 자동화하지 않는 것에 대한 고지:** 네이버 검색 결과 순위나 인스타그램/페이스북 인사이트를 실시간으로 자동 스크래핑하는 기능은 **구현하지 마십시오.** 각 플랫폼 이용약관상 자동 크롤링이 제한되어 있고, 짧은 개발 기간에 안정적으로 만들기도 어렵습니다. 대신 F5(노출 테스트)처럼 "검색 URL을 만들어주고 사람이 확인해 입력하는" 반자동 패턴을 사용하십시오. 이는 기능 축소가 아니라 **의도된 설계**이니 자체적으로 크롤러를 만들려고 시도하지 마십시오.

## 16. 설계 원칙 요약

1. **사람이 하던 진단의 논리를 그대로 코드화** — 임의로 새로운 판정 기준을 만들지 않습니다.
2. **완전 자동화가 가능한 것과 아닌 것을 구분** — 콘텐츠 감사는 완전 자동, 검색 순위·SNS 인사이트는 반자동(사람 확인 + 입력)으로 명확히 나눕니다.
3. **지속 가능한 자체 운영이 목표** — 1회성 데모가 아니라 매달 반복 사용을 전제로 설계합니다.
4. **컨설팅팀의 사전진단 보고서와 같은 언어** — "~로 보입니다" 같은 가설적 어투, 판정과 근거를 함께 제시하는 방식을 그대로 유지합니다.
